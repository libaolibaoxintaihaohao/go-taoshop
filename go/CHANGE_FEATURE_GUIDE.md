# 改项目实战指南

这份文档不讲“怎么从零写项目”，而是专门讲：

- 当你已经有一个项目了
- 现在要加一个新功能
- 应该怎么下手

这里我用一个很典型的需求来讲：

**给用户登录增加验证码功能**

这个需求很适合练手，因为它会逼你同时改：

- 前端
- handler
- service
- 可能的缓存逻辑
- 接口参数校验

这正是“改项目”的核心能力。

## 1. 先明白：改项目不是直接写代码

很多人第一次改项目时，最大的误区就是：

- 一上来就打开某个文件开始写

这通常会越改越乱。

更正确的顺序是：

1. 先明确需求
2. 再判断这个需求会影响哪几层
3. 再决定改动顺序
4. 最后才开始写代码

你要记住一句话：

**改项目，先改脑子里的调用链，再改代码。**

## 2. 需求拆解：什么叫“给用户增加验证码”

这个需求表面很简单，但你要先问清楚是哪一种验证码。

验证码常见有两类：

### 方案 A：图形验证码

比如：

- 登录页先展示一张验证码图片
- 用户输入图片里的字符

### 方案 B：邮箱 / 手机验证码

比如：

- 用户登录前先点“发送验证码”
- 再输入收到的 6 位码

对于你当前这个小淘宝项目，**最适合先练的是图形验证码的简化版**。

为什么：

- 不需要接短信或邮箱服务
- 不需要额外外部平台
- 更适合本地练手

所以这份文档先按这个目标来讲：

**给登录功能增加一个简单验证码校验机制**

## 3. 先判断这个需求会改哪些文件

你这个项目的登录链路大致是：

前端登录表单 -> [app.js](C:\Users\HP\Desktop\go_test\web\assets\app.js) -> [auth_handler.go](C:\Users\HP\Desktop\go_test\internal\http\handler\auth_handler.go) -> [auth_service.go](C:\Users\HP\Desktop\go_test\internal\service\auth_service.go) -> [user_repository.go](C:\Users\HP\Desktop\go_test\internal\repository\user_repository.go)

如果给登录增加验证码，这个需求通常会影响：

1. 前端登录表单
2. 登录请求参数结构体
3. 登录 handler
4. 验证码 service
5. 可选：Redis 缓存

一般 **不需要改数据库表**。

所以你脑子里应该先有这张图：

```text
新增验证码接口
前端先获取验证码
用户输入验证码
登录接口多带一个 captcha_id + captcha_code
后端先校验验证码
验证码通过后再走原来的账号密码登录逻辑
```

## 4. 改项目时，最推荐的顺序

如果是这个需求，我建议按下面顺序改：

1. 先设计接口
2. 再加后端验证码逻辑
3. 再改登录接口参数
4. 再改前端表单
5. 最后联调测试

为什么不是先改前端？

因为前端最终是要配合后端接口的。
后端接口没想清楚，前端很容易反复改。

## 5. 第一步：先设计新增接口，而不是急着写代码

你先要设计：

### 新增一个接口

例如：

- `GET /api/auth/captcha`

它返回：

- `captcha_id`
- `captcha_text` 或图片内容

如果你只是本地练手，可以先不做图片，先直接返回一段文本验证码，方便你调通流程。

例如：

```json
{
  "captcha_id": "abc123",
  "captcha_value": "7K9P"
}
```

真实项目里不应该把答案直接返回给前端，但练手阶段可以先这么做，等你流程跑通后再升级成图片或 SVG。

### 登录接口增加字段

原来登录请求大概是：

```json
{
  "email": "test@example.com",
  "password": "123456"
}
```

改完后变成：

```json
{
  "email": "test@example.com",
  "password": "123456",
  "captcha_id": "abc123",
  "captcha_code": "7K9P"
}
```

这一步非常重要。

因为你改项目时，第一件事不是写代码，而是先把：

- 输入长什么样
- 输出长什么样

想清楚。

## 6. 第二步：先想“验证码存哪”

验证码不能只生成出来，还得有地方保存，后面登录时才能校验。

这里通常有几种方式：

### 方案 1：存内存

简单，但服务重启就丢。

### 方案 2：存 MySQL

能做，但太重了，不适合这种短生命周期数据。

### 方案 3：存 Redis

最常见，也最适合。

对于你这个项目，最推荐：

- `captcha_id` 作为 key 的一部分
- 验证码文本作为 value
- 设置 2 到 5 分钟过期时间

例如：

```text
captcha:abc123 -> 7K9P
```

所以你会自然想到：

- 这个功能应该放在 service 层
- 它底层可能依赖 Redis

## 7. 第三步：先决定新增哪些代码层

对于这个需求，你大概率需要新增：

### 一个 captcha service

比如：

- `internal/service/captcha_service.go`

负责：

- 生成验证码
- 写入 Redis
- 校验验证码

### 一个 captcha handler

如果你想职责清晰，可以新增：

- `internal/http/handler/captcha_handler.go`

负责：

- 提供获取验证码接口

也可以先不单独拆 handler，而是放在 auth handler 里。

如果你是练手阶段，我建议：

- 可以先放在 `auth_handler.go`

这样改动更小，容易跑通。

## 8. 第四步：真实改动路径应该是什么

下面按你这个项目的结构来讲。

### 第 1 处：改配置或复用 Redis

你的项目已经有 Redis 连接了：

- [redis.go](C:\Users\HP\Desktop\go_test\internal\cache\redis.go)

所以这里通常不需要新建 Redis 连接。

你只需要让验证码 service 能拿到已有的 `redis.Client`。

### 第 2 处：新增 service

你可以新增一个 service，比如：

- `CaptchaService`

它至少要有两个方法：

- `Generate(ctx)`  
  生成验证码，保存到 Redis，返回 `captcha_id` 和验证码内容

- `Verify(ctx, captchaID, captchaCode)`  
  校验验证码是否正确

### 为什么先写 service

因为验证码真正的核心逻辑是：

- 生成
- 存储
- 校验

这属于业务逻辑，不属于 handler。

### 第 3 处：改 `auth_handler.go`

文件：

- [auth_handler.go](C:\Users\HP\Desktop\go_test\internal\http\handler\auth_handler.go)

你需要做两类改动：

#### 改动 A：给登录请求结构体增加字段

现在的 `loginRequest` 只有：

- `Email`
- `Password`

你需要新增：

- `CaptchaID`
- `CaptchaCode`

#### 改动 B：在 `Login` 里先校验验证码

顺序变成：

1. 绑定 JSON
2. 调用 `captchaService.Verify(...)`
3. 验证码通过，才继续走 `authService.Login(...)`

### 为什么在 handler 层先校验

因为验证码本身就是登录接口的一部分输入校验。

你也可以把它放进 auth service，但对初学者来说，先在 handler 里串起来更容易理解。

## 9. 第五步：改 `server.go` 做依赖注入

文件：

- [server.go](C:\Users\HP\Desktop\go_test\internal\app\server.go)

如果你新增了 `CaptchaService`，那就要在这里把它创建出来，再传给 handler。

也就是说，`server.go` 的作用会再次体现出来：

- 它不是业务逻辑
- 它是装配中心

你会在这里做类似这样的动作：

1. 用现有 Redis 创建 `CaptchaService`
2. 创建新的 `AuthHandler` 时，把 `captchaService` 一起传进去
3. 注册新的验证码接口

## 10. 第六步：再改前端

文件：

- [index.html](C:\Users\HP\Desktop\go_test\web\index.html)
- [app.js](C:\Users\HP\Desktop\go_test\web\assets\app.js)

### 你要改什么

#### `index.html`

登录表单增加：

- 一个显示验证码的区域
- 一个输入验证码的输入框

#### `app.js`

增加：

- 页面加载时请求 `/api/auth/captcha`
- 保存 `captcha_id`
- 登录时把 `captcha_id` 和 `captcha_code` 一起提交

### 为什么前端放后面改

因为当前端开始改时，你已经知道：

- 后端需要什么字段
- 后端返回什么数据

这样最稳。

## 11. 第七步：最后做联调

联调时你应该按下面顺序测：

1. 单独测验证码接口是不是能拿到数据
2. 测验证码错误时登录是否被拒绝
3. 测验证码正确时是否还能正常登录
4. 测验证码是否过期
5. 测验证码是否能重复使用

### 为什么最后两个很重要

因为验证码的核心安全点不是“能不能生成”，而是：

- 会不会过期
- 能不能重复用

如果验证码能无限次重复用，那安全性就很差。

## 12. 你改项目时最容易卡住的地方

### 卡点 1：不知道从哪层开始

解决办法：

先从“输入输出设计”开始，再往后推。

### 卡点 2：总想一步到位

比如一开始就想做图片验证码、SVG、前端刷新、大小写不敏感、失败次数限制。

这会让你很难推进。

更好的做法是：

先做一个最小可运行版本。

例如：

- 先返回明文验证码
- 先把流程跑通
- 再升级

### 卡点 3：不知道哪些层该改

你可以问自己 3 个问题：

1. 这个需求有没有新增输入输出？
   有的话通常要改 handler / 前端

2. 这个需求有没有新增业务规则？
   有的话通常要改 service

3. 这个需求有没有新增数据存取？
   有的话通常要改 repository / cache / database

## 13. 给你一个最实用的“改项目模板”

以后你改任何功能，都可以按这个模板走。

### 第一步：一句话写需求

例如：

- 给登录增加验证码

### 第二步：写输入输出变化

例如：

- 新增 `GET /api/auth/captcha`
- 登录多传 `captcha_id` 和 `captcha_code`

### 第三步：列受影响层

例如：

- 前端
- handler
- service
- Redis

### 第四步：先改后端，再改前端

顺序建议：

1. service
2. handler
3. router / server 装配
4. 前端
5. 联调

### 第五步：最后补测试点

例如：

- 正确验证码能登录
- 错误验证码不能登录
- 过期验证码不能登录
- 用过一次后不能复用

## 14. 如果你真的要动手改，我建议你怎么练

你可以分 3 轮练。

### 第一轮：最小版

- 验证码接口直接返回明文
- Redis 存 5 分钟
- 登录时校验一次

目标：

- 把调用链跑通

### 第二轮：正式一点

- 验证码只返回 ID，不直接返回答案
- 前端改成图片或 SVG
- 验证成功后立刻删除 Redis key

目标：

- 更接近真实项目

### 第三轮：增强版

- 增加验证码刷新
- 增加错误次数限制
- 增加大小写不敏感校验

目标：

- 练需求扩展能力

## 15. 最后给你一句最关键的话

你现在觉得“改项目难”，不是因为你不行，而是因为你还没有建立一个固定的改动套路。

真正会改项目的人，不是一下子想出全部代码，而是脑子里会自动分这几步：

- 需求改了什么
- 哪几层会受影响
- 先改哪层最稳
- 怎么验证改对了

只要你按这套方式练几次，改项目会比你现在轻松很多。
